# Phase 4: Prediction API - Research

**Researched:** 2026-03-16
**Domain:** FastAPI REST API serving pre-computed NFL game predictions with model hot-reload
**Confidence:** HIGH

## Summary

Phase 4 builds a FastAPI prediction service with 6 endpoints (5 from CONTEXT.md decisions + 1 health check) that serve pre-computed predictions from a PostgreSQL `predictions` table. The model (XGBoost, saved as JSON via `model.save_model()`) is loaded at startup and can be hot-swapped via `POST /model/reload`. Predictions are generated by `models/predict.py` (not by the API per-request), keeping prediction logic with model code.

The core technical challenge is the prediction generation pipeline in `models/predict.py`: it must load the trained model, build features for unplayed games using `features/build.py`, run `model.predict_proba()`, and store results in a new `predictions` table. The API itself is straightforward CRUD-read + one write endpoint. A secondary concern is verifying that `features/build.py` can handle games where `home_score IS NULL` (future/unplayed games); current data has no NULL-score games so this path is untested.

**Primary recommendation:** Use FastAPI with lifespan context manager for model loading, Pydantic v2 response models for all endpoints, and `httpx` + `TestClient` for testing. Keep the API as a thin read layer over PostgreSQL + `experiments.jsonl`. Predictions are pre-computed by `models/predict.py` and stored in a new `predictions` table.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- Pre-compute and store: `models/predict.py` runs after weekly refresh, writes predictions to a `predictions` DB table. API reads from the table.
- POST /model/reload both swaps the in-memory model AND re-generates predictions for the current week with the new model.
- Predictions table keyed by game_id only (no model version tracking). Regenerate replaces existing predictions for that week.
- Prediction logic lives in `models/predict.py` (not in api/), keeping prediction code with model code.
- Auto-detect current week from schedule: query for earliest week in current season where `home_score IS NULL`.
- Current season = `MAX(season)` from `raw_schedules` table. Data-driven, not calendar-based.
- GET /predictions/current is a convenience endpoint that auto-resolves to the detected current week.
- Offseason behavior: return `{"status": "offseason", "predictions": []}` when no unplayed games exist.
- Prediction responses include full game metadata: home_team, away_team, game_date, week.
- Predictions return predicted_winner + confidence (single number), NOT both teams' probabilities.
- API returns a confidence_tier field (high/medium/low) alongside raw confidence. Tier thresholds configurable in config.py.
- GET /model/info returns: experiment ID, training date, 2023 val accuracy, feature count, experiment description, AND both baseline accuracies.
- /predictions/week/{week} accepts optional `?season=` query param. Defaults to current season.
- /predictions/history accepts optional `?season=` and `?team=` query params. Defaults to current season.
- /predictions/history response includes a summary object (`{correct, total, accuracy}`) alongside the predictions array.
- Prediction correctness (actual_winner, correct columns) filled by the weekly refresh pipeline -- not computed by the API on-the-fly.
- POST /model/reload requires `X-Reload-Token` header checked against `RELOAD_TOKEN` env var. Rejects with 403.
- GET /api/experiments reads and parses `experiments.jsonl` directly on each request. File is the single source of truth.
- GET /api/health included for Docker health checks.
- Current week detection must query schedule for next unplayed week -- do NOT hardcode week numbers.
- features/build.py needs to handle games where home_score IS NULL. Verify during implementation.

### Pattern 1: Lifespan Model Loading
**What:** Load XGBoost model and experiment metadata at app startup via async context manager.
**When to use:** Always -- model must be loaded once, not per-request.
**Example:**
```python
# Source: FastAPI official docs (https://fastapi.tiangolo.com/advanced/events/)
from contextlib import asynccontextmanager
from fastapi import FastAPI
from xgboost import XGBClassifier

app_state = {}

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup: load model and metadata
    model = XGBClassifier()
    model.load_model("models/artifacts/best_model.json")
    app_state["model"] = model
    app_state["model_info"] = _load_best_experiment_metadata()
    yield
    # Shutdown: cleanup
    app_state.clear()

app = FastAPI(lifespan=lifespan)
```

### Pattern 2: Pre-computed Predictions (models/predict.py)
**What:** Separate prediction generation from API serving. `models/predict.py` builds features, runs inference, writes to DB.
**When to use:** After model reload, after weekly refresh.
**Example:**
```python
# models/predict.py
def generate_predictions(model, season, week, engine):
    """Generate predictions for a specific week and store in DB."""
    # 1. Build features for the target week
    features_df = build_game_features(seasons=[season])
    week_games = features_df[features_df["week"] == week]

    # 2. Run model inference
    feature_cols = model.get_booster().feature_names
    X = week_games[feature_cols]
    proba = model.predict_proba(X)[:, 1]  # P(home_win)

    # 3. Determine predicted winner and confidence
    for idx, row in week_games.iterrows():
        home_prob = proba[idx_position]
        if home_prob >= 0.5:
            winner = row["home_team"]
            confidence = home_prob
        else:
            winner = row["away_team"]
            confidence = 1.0 - home_prob
        # 4. Upsert to predictions table
```

### Pattern 3: Current Week Detection
**What:** Query schedule for the earliest week with NULL `home_score` in the latest season.
**When to use:** GET /predictions/current, POST /model/reload.
**Example:**
```python
def detect_current_week(engine):
    """Data-driven current week detection."""
    query = """
    SELECT MIN(week) as current_week, MAX(season) as current_season
    FROM schedules
    WHERE season = (SELECT MAX(season) FROM schedules WHERE game_type = 'REG')
      AND game_type = 'REG'
      AND home_score IS NULL
    """
    result = pd.read_sql(query, engine)
    if result["current_week"].iloc[0] is None:
        return None  # offseason
    return result["current_season"].iloc[0], result["current_week"].iloc[0]
```

### Pattern 4: Reload Token Protection
**What:** Simple shared-secret header check for POST /model/reload.
**When to use:** Only on the reload endpoint.
**Example:**
```python
from fastapi import Header, HTTPException

async def verify_reload_token(x_reload_token: str = Header(...)):
    expected = os.environ.get("RELOAD_TOKEN", "")
    if not expected or x_reload_token != expected:
        raise HTTPException(status_code=403, detail="Invalid reload token")
```

### Anti-Patterns to Avoid
- **Per-request model loading:** Loading XGBoost model on every request adds ~50ms+ latency. Load once at startup, swap on reload.
- **Computing predictions on-the-fly:** The CONTEXT.md explicitly requires pre-computed predictions. Don't run `predict_proba()` in the API request handler.
- **Hardcoding week numbers:** Use `detect_current_week()` to query the schedule. The current week changes every Tuesday during the season.
- **Computing correctness in the API:** `actual_winner` and `correct` columns in the predictions table are filled by the weekly refresh pipeline (Phase 6), not computed on-the-fly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| API input validation | Manual if/else checks | Pydantic models + FastAPI query params | Automatic validation, error messages, OpenAPI docs |
| CORS handling | Custom middleware | `fastapi.middleware.cors.CORSMiddleware` | Battle-tested, correct headers for preflight |
| API documentation | Swagger/OpenAPI spec files | FastAPI auto-generated docs | `/docs` and `/redoc` come free |
| Test HTTP client | `requests` library | `fastapi.testclient.TestClient` | Handles lifespan events, no server needed |
| Model serialization | pickle/joblib | XGBoost native JSON (`save_model`/`load_model`) | Already used in Phase 3; portable, version-safe |

**Key insight:** FastAPI eliminates most boilerplate. The API code should be thin -- most logic lives in `models/predict.py` and database queries.

## Common Pitfalls

### Pitfall 1: Features for Unplayed Games
**What goes wrong:** `features/build.py` currently processes all seasons 2005-2024, where every game has results. When predicting future games (home_score IS NULL), `aggregate_game_stats()` uses `result` column which will be NaN/NULL.
**Why it happens:** The `win` and `point_diff` calculations in `aggregate_game_stats()` depend on the `result` column: `home['point_diff'] = home['result'].astype(float)` and `home['win'] = home['result'].apply(lambda x: 1.0 if x > 0 else ...)`.
**How to avoid:** `models/predict.py` should NOT try to include the unplayed game in the rolling feature computation. It should build features from all COMPLETED games, then look up the rolling stats for each team as of the last completed game. The schedule provides `home_rest`, `away_rest`, `div_game` for the future game. The rolling features carry forward from the team's last played game.
**Warning signs:** NaN values in feature columns when predicting. The model will fail or produce incorrect results.

### Pitfall 2: Confidence Score Interpretation
**What goes wrong:** Confusing the raw `predict_proba()[:, 1]` output (P(home_win)) with "confidence in the predicted winner."
**Why it happens:** `predict_proba()[:, 1]` returns the probability of class 1 (home_win). If this is 0.35, the model predicts the away team wins with 65% confidence -- but the raw value is 0.35, not 0.65.
**How to avoid:** Confidence = `max(home_prob, 1 - home_prob)`. If home_prob >= 0.5, predicted_winner = home_team, confidence = home_prob. Otherwise, predicted_winner = away_team, confidence = 1 - home_prob.
**Warning signs:** All predictions showing low confidence even when the model is confident about the away team winning.

### Pitfall 3: predictions Table Schema Mismatch with CONTEXT.md
**What goes wrong:** The ARCHITECTURE.md schema has `PRIMARY KEY (game_id, model_id)` but CONTEXT.md says "keyed by game_id only (no model version tracking)."
**Why it happens:** ARCHITECTURE.md was written before the CONTEXT.md discussion locked this decision.
**How to avoid:** Follow CONTEXT.md. Use `PRIMARY KEY (game_id)` only. This means regenerating predictions replaces the existing row via upsert.

### Pitfall 4: Stale Model Metadata After Reload
**What goes wrong:** After `POST /model/reload`, the model is swapped but `/model/info` still returns old metadata because the experiments.jsonl wasn't re-parsed.
**Why it happens:** Model metadata is cached at startup but not refreshed on reload.
**How to avoid:** The reload handler must: (1) load new model, (2) re-parse experiments.jsonl for the best kept experiment, (3) update cached metadata, (4) re-generate predictions.

### Pitfall 5: TestClient Without Lifespan
**What goes wrong:** Tests that create `TestClient(app)` without considering that the lifespan context manager needs to run, causing `KeyError` when accessing `app_state["model"]`.
**Why it happens:** TestClient does execute lifespan events by default, but the model file must exist at the expected path during tests.
**How to avoid:** Use dependency injection so tests can override the model dependency. Or use a test fixture that creates a temporary model file.

## Open Questions

1. **Feature generation for unplayed games**
   - What we know: `build_game_features()` works for completed games. Rolling stats use `shift(1) + expanding().mean()` which handles NaN gracefully. Schedule provides `home_rest`, `away_rest`, `div_game` for all games.
   - What's unclear: When predicting a future game, `aggregate_game_stats()` tries to compute `point_diff` and `win` from the `result` column which will be NaN. The team won't appear in the team_log for that unplayed game, but their rolling stats from prior games exist.
   - Recommendation: `models/predict.py` should build features for all games in the season, then extract the rolling features from the LAST COMPLETED game for each team. For the target unplayed game, combine: (a) schedule metadata (home_rest, away_rest, div_game), (b) home team's latest rolling stats, (c) away team's latest rolling stats. This may require custom logic rather than reusing `build_game_features()` directly, or filtering the output after building.

2. **Confidence tier thresholds**
   - What we know: The model's best experiment has P(home_win) centered around 0.5 with XGBoost binary logistic output. The actual distribution needs observation.
   - What's unclear: What threshold values produce a useful distribution of high/medium/low predictions.
   - Recommendation: Start with initial defaults (e.g., high >= 0.65, medium >= 0.55, low < 0.55) and make configurable in `api/config.py`. Tune after observing real predictions.
