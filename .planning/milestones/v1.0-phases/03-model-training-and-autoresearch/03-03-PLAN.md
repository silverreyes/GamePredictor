NOTE: These are all drop experiments (one-directional ablation). They test whether removing features improves accuracy, but don't test whether adding features (e.g., different rolling windows, feature interactions) would help. If all ablations hurt accuracy, the conclusion is that the full feature set is already near-optimal — worth noting but not a flaw in the queue.

- [ ] **Exp 4**: Drop situational features (home_rest, away_rest, div_game -- 3 columns removed)
  - Hypothesis: Rest days and divisional flag may add noise without predictive value

- [ ] **Exp 5**: Reduce learning_rate to 0.1, increase n_estimators to 300, add early_stopping_rounds=20
  - Hypothesis: Slower learning with more trees and early stopping prevents overfitting on small dataset
- [ ] **Exp 6**: Reduce max_depth to 3 (from 6)
  - Hypothesis: Shallower trees generalize better on 4000-row dataset
- [ ] **Exp 7**: Set subsample=0.8, colsample_bytree=0.8
  - Hypothesis: Stochastic boosting reduces overfitting via ensemble diversity

### Phase 4: Regularization (Experiments 8-10)
- [ ] **Exp 8**: Increase reg_lambda to 5
  - Hypothesis: L2 regularization constrains leaf weights on small dataset
- [ ] **Exp 9**: Add reg_alpha=0.1 (L1 regularization)

  <action>
Execute the autoresearch experiment loop. This task is fundamentally different from coding tasks -- it is an iterative experimental process.

**Pre-loop setup (before first experiment):**
Plan 03-02 created train.py as a library of functions (load_and_split, train_and_evaluate, log_experiment, etc.) but did NOT create a runnable entry point. Before the first experiment, add a `run_experiment()` function and an `if __name__ == "__main__"` block to train.py that orchestrates the pipeline: load data, split, compute baselines, train, evaluate, log, save. This initial orchestration code is NOT an experiment — it is infrastructure setup. Do not count it as experiment #1. Do not apply keep/revert to this change. Commit it as "feat(03-03): add experiment runner entry point to train.py".

**Loop procedure for each experiment (repeat up to 5 times):**

1. Read models/program.md to identify the next uncompleted experiment
2. Modify models/train.py to implement the experiment (change params dict, feature selection, or hyperparameters in the run_experiment function as specified in the queue)
3. Run the training: execute a Python script that:
   - Calls build_game_features() to load the feature matrix
   - Calls load_and_split() to get temporal splits
   - Calls compute_baselines() to get baseline reference numbers
   - Calls train_and_evaluate() with the experiment's params
   - Calls should_keep() to decide keep vs revert
   - Calls log_experiment() to log to JSONL + MLflow (ALWAYS, regardless of keep/revert)
   - If keep: calls save_model() and save_best_model(), then commits models/train.py with message "exp(03-03): keep exp-{N} -- {hypothesis summary}"
   - If revert: runs `git checkout -- models/train.py` to restore previous version
4. Early termination: if 3 consecutive experiments show <0.3% improvement, stop early per CONTEXT.md termination conditions

**IMPORTANT CONSTRAINTS (from CLAUDE.md):**
- Only modify models/train.py between experiments -- no other production files
- experiments.jsonl is append-only -- log both kept and reverted experiments
- Maximum 5 experiments per session (ceiling, not target)
- Temporal split is hardcoded: train 2005-2022, val 2023, holdout 2024

**Keep/revert decision (from CONTEXT.md locked decision):**
- Keep if accuracy improves by >= 0.5% (>= 0.005 in decimal)
- Keep if accuracy improves by any amount AND log_loss also improves
- Otherwise revert

**At session end:**
- Update models/program.md: check off completed experiments, fill in Current Best table with actual values, add any approaches to Dead Ends section, add session log entry, suggest next 5 experiments
- Commit the final program.md update

**Expected outcome:** XGBoost with 17 features is likely to be competitive with ~60% accuracy. Beating the better-record baseline within 5 experiments is plausible but not guaranteed — the better-record baseline is itself around 60%, so clearing it requires meaningful signal beyond just "pick the team with the better record."

**Contingency if 5 experiments don't beat both baselines:**
If after 5 experiments the best accuracy is below the better-record baseline:
1. Update program.md with results so far and mark this session as "Session 1 — baseline not yet cleared"
2. The checkpoint (Task 3) becomes the decision point: user reviews results, SHAP feature importances, and log_loss trends
3. User can approve a second experiment session (re-run Task 2 with experiments 6-10 from the queue) or reassess the feature set
4. The phase is NOT failed — it means the experiment queue needs more iterations, which is exactly what the multi-session design supports
5. Do NOT artificially inflate results or skip the baseline comparison gate
